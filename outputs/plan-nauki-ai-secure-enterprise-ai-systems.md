# Plan nauki i rozwoju zawodowego: Secure Enterprise AI Systems

## 1. Diagnoza strategiczna

Najważniejszy wniosek: nie startujesz od zera w IT. Masz praktyczne doświadczenie wdrożeniowe, integracyjne i security/operations. Twoja przewaga nie polega dziś na tym, że jesteś klasycznym programistą AI albo data scientistą. Polega na tym, że już pracujesz z klientami, wdrożeniami, SIEM/SOAR, API, konektorami, GitLab Runnerem, Cisco SD-WAN API i technicznym dopinaniem systemów.

To jest dobra baza pod profil:

> **AI Solutions Architect / AI Implementation Lead for secure enterprise AI systems**

Wspierające specjalizacje:

- **AI Security Automation / AI Security Engineer**
- **Enterprise AI Automation / Knowledge Systems Engineer**

Najlepsza robocza tożsamość zawodowa:

> **Projektuję i zabezpieczam systemy AI dla firm.**

To jest lepsze niż: "jestem od promptów", "jestem od Cisco automation", "jestem od local LLM", "jestem czystym RAG specialist" albo "jestem klasycznym AI/ML engineerem".

Twoja ścieżka powinna prowadzić mniej więcej tak:

```text
AI Automation Engineer with Security Focus
    -> AI Security Engineer / Security Automation for AI
    -> AI Solutions Architect / AI Implementation Consultant
```

## 2. Co naprawdę kupuje rynek

Z przeanalizowanych ofert AI Architect, AI Engineer i AI Security wynika, że firmy nie szukają tylko osób od "budowania modeli". Realne problemy biznesowe to:

1. Bezpieczne i kontrolowane wdrażanie AI/GenAI w organizacji.
2. Przejście od eksperymentów, PoC i chatbotów do stabilnych systemów produkcyjnych.
3. Ustandaryzowanie architektury, narzędzi i procesu tworzenia rozwiązań AI.
4. Zmniejszenie ryzyka: security, compliance, RODO, prywatność, dostęp do danych, monitoring.
5. Zwiększenie efektywności zespołów przez automatyzację, CI/CD, wzorce wdrożeniowe i platformy.

Najczęściej powtarzające się bloki kompetencji:

- Python i podstawowy ML/AI stack.
- LLM/GenAI: RAG, agenci, LangChain/LangGraph/LlamaIndex, prompt engineering jako element systemu.
- Cloud: AWS, GCP, Azure, Bedrock, Vertex AI, Azure OpenAI.
- DevOps/MLOps/LLMOps: CI/CD, Docker, Kubernetes, Terraform, IaC, monitoring.
- API i integracje: REST, GraphQL, OpenAPI, API Gateway, mikroserwisy.
- Vector/data layer: vector DB, embeddingi, data pipelines, retrieval storage.
- Governance/security: guardrails, AI Gateway, anonimizacja danych, cloud security, Secure by Design.
- Trade-offy: jakość vs koszt vs opóźnienie, OpenAI vs open source, fine-tuning vs RAG, local vs cloud.

## 3. Docelowy profil zawodowy

### Główna specjalizacja

**Secure Enterprise AI Systems / AI Security & Implementation Architecture**

To oznacza osobę, która potrafi:

- projektować systemy AI/RAG/agentów,
- integrować je z API i systemami firmowymi,
- zabezpieczać dane, dostępy i wywoływanie narzędzi,
- projektować uprawnienia i granice dostępu do wiedzy,
- oceniać ryzyka RODO/security/governance,
- monitorować systemy AI,
- oceniać jakość odpowiedzi,
- tworzyć testy, runbooki i checklisty,
- pisać dokumentację architektoniczną,
- rozumieć wymagania biznesowe,
- prowadzić wdrożenia u klienta lub wewnętrznie,
- łączyć local AI z cloud AI, gdy ma to sens,
- budować portfolio bez ujawniania danych firmowych.

### Specjalizacje wspierające

**1. AI Security Automation**

Bazujesz na SIEM/SOAR, API, konektorach, monitoringach i security operations. Kierunek: prompt injection, data leakage, secure RAG, bezpieczne uprawnienia agentów, audyt, test harnessy, guardrails, AI threat modeling.

**2. Enterprise AI Automation / Knowledge Systems**

Budujesz praktyczne systemy: RAG, agenci, przepływ pracy automation, integracje z API, routing zgłoszeń, klasyfikacja dokumentów, systemy wiedzy, evals, monitoring, runbooki.

**3. Cloud AI jako filar, nie główna tożsamość**

Cloud AI jest ważne, bo najbardziej rynkowe projekty będą często hybrydowe: część lokalnie dla danych wrażliwych, część w chmurze dla skalowania, governance, usług managed i szybkości delivery.

- vibe codingu jako tożsamości zawodowej,
- klasycznego data science bez mocnego powodu biznesowego,
- trenowania modeli od zera,
- klasycznego AI/ML researchu,
- czystego backend developmentu,
- czystego Cisco/network automation,
- czystego RAG specialist,
- czystego local LLMOps engineer,
- klasycznego pentestingu/red teamingu jako jedynej ścieżki,
- losowych narzędzi AI bez odpowiedzialności wdrożeniowej.

Uzasadnienie: te kierunki są albo słabo dopasowane do Twoich preferencji, albo mają dłuższy time-to-value, albo mogą Cię zamknąć w niszy o niższym potencjale B2B i konsultingowym.

## 5. AUTOCOR / CCNP Automation

AUTOCOR traktuj jako tor taktyczny.

### Decyzja

- **AUTOCOR: tak.**
- **ENAUTO: nie teraz.**
- **DCNAUTO: nie teraz.**
- **Cisco jako główna ścieżka: nie.**

AUTOCOR ma sens, bo:

- może pomóc w obecnej firmie,
- może pomóc w podwyżce,
- może dać dostęp do ciekawszych zadań automatyzacyjnych,
- rozwija API, GitLab, Ansible, Terraform, telemetry, automatyzację i myślenie systemowe,
- wzmacnia Twoją wiarygodność jako osoby od automatyzacji.

Nie powinien jednak stać się Twoją tożsamością na 5 lat. Cisco automation niesie ryzyko utknięcia w integratorskich wdrożeniach sieciowych, presji poza godzinami i pracy nocnej.

### Ile czasu tygodniowo

Przy limicie 14 h tygodniowo:

- przez pierwsze 3-4 miesiące: **5-6 h tygodniowo na AUTOCOR**, bo początkowo jest głównym celem taktycznym,
- równolegle Python jako drugi najważniejszy tor: **3-4 h tygodniowo** przez DataCamp, coddy.tech i małe laboratoria,
- po zdaniu AUTOCOR: **0-1 h tygodniowo tylko utrzymanie i zadania z pracy**,
- nie rozwijać głębiej Cisco, chyba że firma jasno pokaże znaczący wzrost wynagrodzenia lub wyjątkowo dobre projekty.

Monitoring:

```text
Prometheus
 -> node_exporter
```

## 19. Ryzyka i zabezpieczenia

Zabezpieczenie: AUTOCOR tak, potem stop. Cisco jako narzędzie, nie tożsamość.

### Ryzyko: zbyt akademickie AI

Zabezpieczenie: każda teoria musi trafić do projektu, testu, checklisty lub decyzji architektonicznej.

### Ryzyko: zbyt powierzchowne AI

Zabezpieczenie: nie kończyć na promptach i demo. W każdym projekcie dodawać security, evals, monitoring, testy i runbook.

### Ryzyko: zbyt wąski RAG

Zabezpieczenie: RAG traktować jako część enterprise knowledge systems i secure AI architecture.

### Ryzyko: za mało angielskiego

Zabezpieczenie: każdy projekt publiczny po angielsku, regularne nagrania i symulacje rozmów.

### Ryzyko: brak portfolio

Zabezpieczenie: mierzyć postęp przez artefakty, nie kursy.

### Ryzyko: za dużo pracy bez wpływu na CV

Zabezpieczenie: każde większe zadanie w pracy przekładać na anonimowy opis kompetencji, ADR, checklistę lub studium przypadku.

### Ryzyko: AI red teaming only

Zabezpieczenie: AI security łączyć z architecture, automation i implementation.
